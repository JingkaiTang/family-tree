# 家族树布局引擎重新设计

**日期**: 2026-05-24
**状态**: 草稿

## 概述

两个独立功能：
1. **优化现有自动排版** — 用 ELK.js 约束布局引擎替换手工 BFS 算法，解决重叠和位置不合理问题
2. **主角视角布局** — 新增以选中人物为中心的树状辐射布局模式

## 功能 1：ELK.js 约束布局引擎

### 目标

- 消除节点重叠
- 新增亲属自动放置在合理位置
- 保持辈分结构（从上到下）
- 配偶同层并排
- 兄弟姐妹紧凑排列

### 技术选型

使用 `elkjs`（Eclipse Layout Kernel 的 JS 版本）。

理由：
- 成熟的层次化图布局库
- 原生支持节点防重叠（硬约束）
- 支持父节点在子节点上方（层次约束）
- 支持同层对齐和间距控制
- 纯前端，无后端依赖

### 接口设计

保持现有接口不变：

```typescript
export function layoutFamilyTree(
  members: Member[],
  opts?: { manualPositions?: Record<string, { cx: number; top: number }> },
): LayoutResult
```

输入输出完全兼容，调用方无需修改。

### 布局策略

```
ELK 配置:
- algorithm: layered
- direction: DOWN (从上到下)
- nodePlacement: BRANDES_KOEPF (平衡布局)
- edgeRouting: ORTHOGONAL (直角连线)
- spacing.nodeNode: 自动计算 (消除重叠)
- layering.strategy: NETWORK_SIMPLEX (最优层次分配)
```

### 配偶处理

将一对配偶合并为一个"虚拟节点"参与 ELK 布局：
- ELK 计算虚拟节点位置后，内部拆分为两个真实节点
- 配偶之间用水平短线连接

### 兄弟姐妹处理

同一父母的子女设置 `alignmentConstraint`，保证：
- 同代同行
- 紧凑排列（间距最小化）

### 独生子女对齐

使用 ELK 的 `alignmentConstraint`，让独生子女的 x 坐标 = 父母中点 x。

### 手工位置覆盖

保持现有逻辑：`manualPositions` 中的节点在 ELK 布局后覆盖坐标。

### 连线

ELK 的 `edgeRouting: ORTHOGONAL` 自动生成直角连线，替代现有手工 polyline 逻辑。

| `src/core/treeLayout.ts` | 重写为 ELK 调用，保持接口 |

## 功能 2：主角视角混合布局

### 目标

- 以选中人物为中心，保留辈分树结构
- 关系近的人在身边，关系远的人在外围
- 沿实际血缘/姻亲树向外生长
- 与现有"视角"功能（计算称谓）明确区分

### 布局结构

以主角为根，沿家族树拓扑向外辐射：

```
            [爷爷]——[奶奶]    [外公]——[外婆]
               |                  |
          [爸爸]——[妈妈]———[舅舅]
               |
  [配偶]——[ 我 ]——[兄弟]
               |
            [孩子]
```

### 核心算法

1. **关系距离计算**
   - 从主角出发 BFS，计算到每个人的最短关系步数
   - 配偶 = 1步，父母/子女 = 1步，兄弟姐妹 = 2步，祖辈 = 2步，堂表 = 3步

2. **树结构构建**
   - 以主角为根，构建两棵树：
     - 主角的血缘树（主角的祖先 + 后代 + 兄弟姐妹）
     - 配偶的血缘树（配偶的祖先 + 兄弟姐妹）
   - 两棵树通过主角-配偶连线合并
   - 多配偶情况：每个配偶独立一棵血缘树，均连接到主角

3. **坐标分配**
   - 每代一个水平行（与现有布局方向一致）
   - 主角行在画布中心
   - 上方：主角的父母 → 祖父母...
   - 下方：主角的子女 → 孙辈...
   - 配偶的亲属在配偶侧展开

4. **间距控制**
   - 直系亲属（父母/子女/配偶/兄弟姐妹）：节点间距 = 1.5 单位
   - 旁系亲属（叔伯/堂表）：节点间距 = 2.5 单位
   - 远亲：节点间距 = 3.5 单位
   - 代间距保持现有 ROW_HEIGHT = 7 单位

### UI 设计

**新增按钮**："以选中为中心布局"

位置：在现有"以选中为视角"按钮旁边

行为：
- 选中某成员后，点击按钮切换到主角视角布局
- 布局以该成员为中心重新排列
- 再次点击同一按钮 → 退出主角视角，回到传统布局
- 选中不同成员后点击按钮 → 切换到新成员为中心的布局

**状态管理**：
- UI store 新增 `centerLayoutId: string | null`
- 与现有 `viewpointId` 完全独立
- `viewpointId` = 计算称谓用
- `centerLayoutId` = 控制布局模式

## 实施顺序

1. 先实现功能 1（ELK 布局引擎）— 这是基础，影响所有布局
2. 再实现功能 2（主角视角）— 依赖功能 1 的布局能力

## 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| ELK.js 包体积大（~200KB） | 动态 import，按需加载 |
| ELK 布局结果不如手工调优 | 保留 manualPositions 覆盖能力 |
| 主角视角在大家族时图太大 | 限制显示层级，提供缩放控制 |
