## 概述

两个独立功能：
1. **优化现有自动排版** — 用 ELK.js 约束布局引擎替换手工 BFS 算法，解决重叠和位置不合理问题
2. **主角视角布局** — 新增以选中人物为中心的树状辐射布局模式

## 功能 1：ELK.js 约束布局引擎

### 目标

- 消除节点重叠
- 新增亲属自动放置在合理位置
- 保持辈分结构（从上到下）
- 配偶同层并排
- 兄弟姐妹紧凑排列

### 技术选型

使用 `elkjs`（Eclipse Layout Kernel 的 JS 版本）。

理由：
- 成熟的层次化图布局库
- 原生支持节点防重叠（硬约束）
- 支持父节点在子节点上方（层次约束）
- 支持同层对齐和间距控制
- 纯前端，无后端依赖

### 接口设计

保持现有接口不变：

```typescript
export async function layoutFamilyTree(
  members: Member[],
  opts?: { manualPositions?: Record<string, { cx: number; top: number }> },
): Promise<LayoutResult>
```

输入输出完全兼容，调用方无需修改。

### 布局策略

```
ELK 配置:
- algorithm: layered
- direction: DOWN (从上到下)
- nodePlacement: BRANDES_KOEPF (平衡布局)
- edgeRouting: ORTHOGONAL (直角连线)
- spacing.nodeNode: 自动计算 (消除重叠)
- layering.strategy: NETWORK_SIMPLEX (最优层次分配)
```

### 配偶处理

将一对配偶合并为一个"虚拟节点"参与 ELK 布局：
- ELK 计算虚拟节点位置后，内部拆分为两个真实节点
- 配偶之间用水平短线连接

### 兄弟姐妹处理

当前实现没有使用 ELK 的 `alignmentConstraint` API，而是通过辈分推导、夫妻虚拟节点、父母到子女虚拟节点的边，以及测试约束保证：
- 同一父母下的子女在同代
- 多兄弟姐妹不会被其他 couple 插入
- 整体节点不重叠

### 独生子女对齐

在 ELK 布局完成后执行后处理：如果某个 couple 只有一个子女，且该子女的父母集合与 couple 匹配，则把子女 `cx` 对齐到父母中点；手工位置覆盖优先级更高。

### 手工位置覆盖

保持现有逻辑：`manualPositions` 中的节点在 ELK 布局后覆盖坐标。

### 连线

ELK 负责节点位置；连线仍由 `elkLayout.ts` 手工生成 polyline。这样可以保留夫妻线、父母-子女折线、干亲虚线和手工拖动后的端点同步。

### 文件变更

| 文件 | 变更 |
|------|------|
| `package.json` | 添加 `elkjs` 依赖 |
| `src/core/treeLayout.ts` | 重写为 ELK 调用，保持接口 |
| `src/core/treeLayout.test.ts` | 更新测试用例 |

## 功能 2：主角视角混合布局

### 目标

- 以选中人物为中心，按关系距离形成可读层次
- 关系近的人在身边，关系远的人在外围
- 沿父母、子女、配偶、兄弟姐妹、干亲等关系向外扩展
- 与现有"视角"功能（计算称谓）明确区分

### 布局结构

当前主角视角采用“关系距离环形布局”，不是传统上下代树形布局：
- 第 0 层：主角，放在画布中心
- 第 1 层：父母、子女、配偶、干亲等一步关系
- 第 2 层：祖辈、孙辈、兄弟姐妹等两步关系
- 更远关系继续向外扩展
- 与主角不连通的成员不会被丢弃，会放在最外层并记录到 `orphanIds`

### 核心算法

1. **关系距离计算**
   - 从主角出发 BFS，计算到每个人的最短关系步数
   - 配偶 = 1步，父母/子女 = 1步，兄弟姐妹 = 2步，祖辈 = 2步，堂表 = 3步

2. **按距离分层**
   - 只把可达关系放入 BFS 距离层
   - 直连兄弟姐妹在缺少父母节点时按 2 步处理
   - 不连通成员归入最外层并标记为 `orphanIds`

3. **坐标分配**
   - 每层先用 ELK 排列，得到稳定的层内顺序
   - 再按距离层映射到同心圆半径
   - 主角平移到画布中心

4. **间距控制**
   - 第 1 层半径为 10 单位
   - 每增加一层，半径增加 8 单位
   - 层内由 ELK 排列并保持节点不重叠

### UI 设计

**新增按钮**："以选中为中心布局"

位置：在现有"以选中为视角"按钮旁边

行为：
- 选中某成员后，点击按钮切换到主角视角布局
- 布局以该成员为中心重新排列
- 进入中心布局后显示"退出中心布局"按钮，点击后回到传统布局
- 选中不同成员后点击按钮 → 切换到新成员为中心的布局

**状态管理**：
- UI store 新增 `centerLayoutId: string | null`
- 与现有 `viewpointId` 完全独立
- `viewpointId` = 计算称谓用
- `centerLayoutId` = 控制布局模式

## 2026-06-25 默认布局方案调整

默认布局不再把所有人节点直接交给通用自由布局处理，而是先构建语义模型：

- PersonNode：真实成员
- UnionNode：父母/配偶/单亲家庭单元
- ChildGroup：同一 Union 下的子女集合
- Component：不连通家族分量

布局目标从“让通用图算法自由寻找坐标”调整为“先满足家谱语义约束，再做稳定排布”。中心布局仍作为独立关系探索模式，不共享默认布局的手工位置持久化语义。
