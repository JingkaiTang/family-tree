# 根族连续网格排版引擎设计

**日期：** 2026-07-15

**状态：** 产品规则已确认，等待正式规格审阅

**关联设计：**

- `2026-07-11-family-unit-grid-layout-design.md`
- `2026-07-15-restore-default-layout-design.md`

## 1. 背景

现有家庭单元网格已经解决夫妻拆分、卡片重叠、线路穿卡和跨家庭共线等基础问题，但仍以整张关系图为一个全局网格进行压缩。同一上层祖辈延伸出的成员虽然通常相邻，却没有稳定、明显的空间归属，多个血缘分支容易在同一代互相穿插。

本设计在家庭单元网格之上增加“根族布局层”：先识别最上层祖辈家庭，再为每个根族分配不可被其他根族插入的连续网格区间。跨根家庭进入根族之间的桥接带；密集跨根关系进入桥接岛。

目标不是给画布增加可见容器，而是通过连续空间、留白、根族色系和线路方向，让用户自然感知不同祖辈根延伸出的家族结构。

## 2. 已确认的产品决策

1. 根是最上层祖辈家庭单元，不是单个人，也不是项目的 `rootMemberId`。
2. 最上层当前夫妻组成一个根家庭；单身祖辈可以独立成为根家庭。
3. 同一根族在所有代际中拥有连续、不可被其他根族插入的网格列区间。
4. 根族之间只使用明显留白，不显示背景色、区域边界或根族标题。
5. 根家庭通过粗色条、较深边框和稍强阴影进行隐式强化，不显示文字或根图标。
6. 同根人物卡使用稳定根族色条；不同家庭线路使用同一根色系的不同变体。
7. 未展开自身祖辈的外来配偶加入伴侣根族，不自动产生新根。
8. 双方都能追溯到不同可见根族时，家庭才是跨根家庭。
9. 两根跨根家庭进入桥接带；密集多根关系进入桥接岛。
10. 跨根夫妻的人物卡按来源根族的空间方向排序，不按性别排序。
11. 跨根家庭的后代形成联合分支，不强行归入父系或母系。
12. 根签名使用集合并集；相同根不会重复计数。
13. 同根不同子分支之间的婚姻仍属于该根族，不创建跨根桥接带。
14. 每个普通家庭、跨根家庭和根家庭默认作为独立家庭单元拖拽；按住 `Ctrl`（macOS 同时支持 `Command`）拖动根家庭时移动整个根族。
15. 所有拖拽只写入默认布局偏好，不修改人物关系、代际或根签名。
16. `rootMemberId` 只决定多个独立家族分量的展示优先级。
17. `viewpointId` 只决定画布聚焦目标。
18. 辅助关系的显示开关不得改变主体根族布局。

## 3. 目标与非目标

### 3.1 目标

- 同一根族的大部分成员形成明显、稳定、紧凑的空间簇。
- 根族在每一代都保持连续，不被其他根族家庭插入。
- 根家庭能够被隐式识别为颜色和分支的起点。
- 跨根家庭的位置同时接近相关根族，且不破坏根族内部连续性。
- 同根家庭线路保持根色亲和，同时仍能分辨不同线路归属。
- 新增祖辈或局部关系后，未受影响根族尽量保持颜色、顺序和位置。
- 相同输入、偏好和上一个场景必须得到确定性输出。
- 默认支持 500 位可见成员。

### 3.2 非目标

- 不显示根族背景面板、边框或标题。
- 不复制人物卡表达多根归属。
- 不按姓氏、性别或父系规则自动决定根族归属。
- 不把 `rootMemberId` 当作唯一布局根。
- 不用任意像素坐标保存拖拽结果。
- 不允许拖拽改变关系事实或根签名。
- 不让历史配偶、继亲、干亲等辅助关系改变主体布局。
- 不追求任意关系图的最小画布面积。

## 4. 第一性原则与约束优先级

家谱中的每个人可以拥有多个上层祖辈，因此“根族”不是生物学上的唯一归属，而是从当前可见关系投影出的布局身份。布局可以选择空间归属，但不能改写关系事实。

约束冲突时使用以下固定优先级：

1. **关系正确：** 人物、配偶和亲子事实不能改变；人物只渲染一次。
2. **几何安全：** 卡片不重叠；线路不穿卡；端点必须连续。
3. **线路归属：** 不同家庭不得共享正长度线段或形成伪汇合。
4. **根族连续：** 单根区域不能被其他根族家庭插入。
5. **桥接语义：** 多根家庭位于桥接区域，不能伪装成单根家庭。
6. **家庭与分支连续：** 夫妻不可拆；同父母子女和直接分支连续排列。
7. **用户布局偏好：** 在不违反前述约束时保留拖拽顺序。
8. **视觉稳定：** 未受影响区域尽量保持原位置和颜色。
9. **紧凑度：** 最后再压缩空列和画布尺寸。

根族留白和画布宽度是有意的产品取舍。不得为了缩窄画布破坏根族连续性。

## 5. 核心模型

### 5.1 根家庭

```ts
interface RootFamily {
  id: string
  rootUnitId: string
  memberIds: string[]
  componentId: string
  generation: number
}
```

根家庭是一个没有更高可见主体父母家庭的最上层 `FamilyUnit`。最上层当前夫妻构成一个根家庭；没有当前配偶的祖辈构成单人根家庭。

### 5.2 根签名

```ts
type RootId = string
type RootSignature = readonly RootId[]
```

根签名必须排序、去重并使用稳定根 ID：

- 单根成员：`['root:A']`
- A、B 联合分支：`['root:A', 'root:B']`
- A、B、C 联合分支：`['root:A', 'root:B', 'root:C']`

根签名是布局派生状态，不写回人物关系。

### 5.3 根族区域与桥接区域

```ts
interface RootDomain {
  id: string
  rootId: RootId
  signature: RootSignature
  columnStart: number
  columnEnd: number
  rootUnitId: string
}

interface BridgeDomain {
  id: string
  signature: RootSignature
  kind: 'pair-band' | 'multi-root-island'
  columnStart: number
  columnEnd: number
  sourceRootIds: RootId[]
}
```

单根签名进入 `RootDomain`。包含两个以上根的签名进入 `BridgeDomain`。

### 5.4 根族关系图

```ts
interface RootInteractionEdge {
  rootIds: RootId[]
  bridgeSignature: RootSignature
  familyUnitIds: string[]
  weight: number
}
```

根族关系图用于确定根族左右顺序。跨根家庭越多、代际越靠上、成员越多，连接权重越高，相关根族越应相邻。

## 6. 根识别

### 6.1 基本规则

1. 先完成主体关系投影和代际分配。
2. 在每个连通家族分量中查找没有主体父母家庭的人物和家庭单元，形成临时来源候选。
3. 当前夫妻双方都是来源候选，且共同形成了独立向下延伸的家庭分支时，合并为一个根家庭。
4. 不要求不同根家庭位于整个分量的同一代；每个分支按自身已录入的最上层祖辈确定根。
5. 如果某个无父母人物的第一个且唯一主体家庭，就是与一个已经拥有上游根签名的人结婚，并且此人没有其他独立向下分支，则把此人判定为未展开祖辈的外来配偶，继承伴侣根族而不产生新根。
6. 其余来源候选成为根；不相连的单人或单家庭分量也必须拥有根。
7. 根识别通过“临时来源 → 签名传播 → 外来配偶抑制”迭代到稳定，不能依赖全局最小 generation。
8. 根家庭 ID 必须从稳定人物 ID 生成，不能依赖数组下标或当前坐标。

例如 A 分支录入了五代祖辈、B 分支只录入两代祖辈，两支后代结婚时，A、B 最上层已录入家庭都应成为根。B 不能因为 generation 比 A 低而被吸收到 A；只有直接出现在 A 后代婚姻中、没有任何独立上游或旁支的配偶才属于“外来配偶”。

### 6.2 动态上移

补录更早祖辈后，新的最上层家庭成为根，原根家庭降为普通后代家庭。布局必须：

- 把原根族成员映射到新根族；
- 通过最大成员重叠继承原根族颜色；
- 尽量保留原根族左右顺序和内部相对顺序；
- 只把根强化样式移动到新的根家庭；
- 不因为根 ID 改变而让整个分支无理由换色或翻转。

### 6.3 `rootMemberId` 的职责

`rootMemberId` 不参与根识别或根签名传播。多个不相连分量存在时：

1. 包含 `rootMemberId` 的分量优先放在整体画布中心。
2. 其他分量按可见成员数量和稳定 ID 排在两侧。
3. 没有 `rootMemberId` 时，成员最多的分量优先。

## 7. 根签名传播

### 7.1 主体关系

只有以下关系参与传播：

- 当前主要配偶关系；
- 主要血缘父母关系；
- 主要收养父母关系；
- `childLayoutAssignments` 选择的主体父母家庭。

以下关系不参与传播：

- 历史配偶；
- 次要父母；
- 未被选为主体的继父母；
- 干亲；
- 其他辅助关系。

### 7.2 传播规则

1. 根家庭获得单根签名 `{A}`。
2. 未展开自身上游根的配偶继承家庭已有签名。
3. 夫妻双方签名相同，家庭保持该签名。
4. 夫妻双方拥有不同可见签名，家庭签名为集合并集。
5. 子女继承主体父母家庭的签名。
6. 收养子女在布局上继承主要收养家庭签名，但关系线保留收养样式。
7. 签名计算直到稳定；相同根只保留一次。

### 7.3 重叠签名

- `{A,B} × {A,B}` 保持 `{A,B}`。
- `{A,B} × {A}` 保持 `{A,B}`，位置偏向 A 根族。
- `{A,B} × {B,C}` 合并为 `{A,B,C}`。
- 同根不同子分支婚姻保持 `{A}`，不创建桥接带。

## 8. 总体布局流水线

```text
FactNormalizer
  → ViewProjector
  → FamilyUnitBuilder
  → GenerationAssigner
  → RootFamilyDiscoverer
  → RootSignaturePropagator
  → RootInteractionGraphBuilder
  → DomainOrderer
  → DomainIntervalAllocator
  → RootBranchPlacer
  → BridgeDomainPlacer
  → RootAwareLaneRouter
  → SceneValidator
```

布局引擎继续保持纯函数：

```ts
function layoutFamilyScene(request: LayoutRequest): LayoutScene
```

根族、桥接区域和路由通道必须作为正式中间模型存在，不能只通过渲染层颜色推测。

## 9. 根族区域布局

### 9.1 连续列强约束

每个 `RootDomain` 拥有跨越所有代际的连续列区间：

- 其他根族和桥接家庭不能进入该区间；
- 同一代该根族的家庭必须连续；
- 空列允许保留；
- 空间不足时扩宽当前根族区间，而不是借用相邻根族空间。

根族间最小留白建议为：

```ts
rootGap = max(cardWidth, familyGap * 2)
```

以当前指标计算，根族间至少保留一个人物卡宽度，并额外满足独立线路通道需求。

### 9.2 根族内部“先分支、后压缩”

1. 从根家庭向下建立主体子女分支。
2. 自底向上计算每个分支需要的连续网格宽度。
3. 自顶向下为分支分配不重叠子区间。
4. 根家庭位于整个根族区间顶部中心。
5. 父母家庭尽量位于直接子女集合中心。
6. 同父母子女家庭连续排列。
7. 夫妻家庭始终作为不可拆单元。
8. 最后只压缩不破坏分支顺序和线路通道的空列。

### 9.3 同根跨分支婚姻

同根两个子分支的成员成为配偶时：

- 家庭保持在根族区间内；
- 放在两个来源子区间之间的最近可用网格；
- 两边父母线分别连接各自人物卡顶部；
- 联合后代从夫妻中心向下；
- 不显示跨根双色标记。

## 10. 根族顺序

根族左右顺序由根族关系图决定：

1. 优先让强连接根族相邻。
2. 最小化加权跨根距离。
3. 最小化跨根线路交叉。
4. 在评分相同时优先保留上一个场景顺序。
5. 再使用用户 `rootOrders`。
6. 最后使用稳定根 ID 打破平局。

没有跨根关系的独立根族之间保留比普通根族间距更大的空白。

## 11. 桥接带与桥接岛

### 11.1 两根桥接带

两个相邻根族之间存在跨根家庭时，创建 `pair-band`：

- 跨根家庭按代际放在两根族之间；
- A 来源人物卡朝向 A 根族，B 来源人物卡朝向 B 根族；
- 同代多个 A×B 家庭横向扩宽桥接带，不重叠堆放；
- 联合后代在桥接家庭下方保持紧凑；
- 根族内部成员不能占用桥接带。

### 11.2 多根桥接岛

当根族关系无法通过一维相邻顺序合理表达，例如同时存在 A×B、B×C、A×C 时，创建 `multi-root-island`：

- 相关跨根家庭在独立连续网格区间内按代际排列；
- 桥接岛没有背景、标题或边框；
- 根族仍分别保持聚合；
- 桥接岛的横向锚点取所有来源根当前位置的算术重心，并限制在最左、最右来源根的跨度内，不能统一追加到组件边缘；
- 各根族通过自己的隐式边界出口连接桥接岛；
- 相同多根签名的后代保持为一个紧凑分支。

### 11.3 多根显示上限

根签名可以包含任意数量的根，但视觉最多直接显示两种主要根色。其余根用 `+N` 小标记表达，避免多色噪声。

## 12. 视觉系统

### 12.1 不使用的元素

- 根族背景色；
- 根族区域边框；
- 根族标题；
- “根”或“支系”文字；
- 常驻根族包围框。

### 12.2 根族层颜色

- 每个根族分配稳定、可区分的主色。
- 根家庭使用 6px 主色色条、较深边框和稍强阴影。
- 普通同根人物使用 3px 主色色条。
- 根动态上移时通过成员重叠继承旧主色。
- 根色分配必须考虑相邻根族色差，不能只对 ID 独立哈希。

### 12.3 家庭层颜色

- 每个家庭从所属根色派生稳定的明暗或饱和度变体。
- 夫妻轴和主体亲子线路使用家庭变体色。
- 不同家庭即使同根，也不能使用完全相同的线路身份。
- 颜色只是辅助信号，线路独占通道仍是硬约束。

### 12.4 跨根家庭

- 双方人物卡保留各自根族色条。
- 夫妻轴从中心分为两种来源根色。
- 家庭单元使用双色细边，不使用双色背景。
- 联合后代使用双短色条。
- 主体亲子线路使用稳定联合色，来源处保留双色标记。

## 13. 根族感知线路路由

### 13.1 固定端口

- 每个人物卡顶部拥有自己的父母入线端口。
- 夫妻中心 `UnionHub` 是共同子女的唯一主体出线端口。
- 跨根双方的上游父母线分别连接各自人物卡，不能连接夫妻中心。

### 13.2 根族内部通道

- 路线优先使用家庭分支子区间内的专属通道。
- 同一 `ParentageGroup` 的多名子女可以共享主干和 bus。
- 不同 `routeOwnerId` 不得共享任何正长度水平或垂直线段。
- 根族内部线路不能借用其他根族或桥接区域绕行。
- 无路可走时扩宽本根族区间并重新路由。

### 13.3 隐式边界出口

每个根族为每个相邻桥接区域分配不可见的边界出口：

- 出口只存在于路由模型，不渲染圆点、文字或标记；
- 根族内部线路先到达出口，再进入桥接通道；
- 不同来源家庭不能共用出口之后的线路段；
- 桥接带为两侧根族保留独立入口通道。

### 13.4 交叉与伪汇合

- 不同家庭线路可以点交叉。
- 交叉处使用 line bridge 表示跨越。
- 禁止不同家庭形成 T 形接点。
- 交叉点与弯点保持最小安全间距。
- 线路 z-order 和桥方向必须确定性输出。

## 14. 拖拽与布局偏好

### 14.1 偏好模型

```ts
interface RootOrderPreference {
  componentId: string
  rootIds: string[]
}

interface RowOrderPreference {
  domainId: string
  generation: number
  unitIds: string[]
  columns?: Record<FamilyUnitId, number>
}

interface BridgeOrderPreference {
  bridgeDomainId: string
  generation: number
  unitIds: string[]
}
```

`columns` 保存家庭单元相对所属 domain 左侧的网格列，不保存任意像素坐标，也不允许改变代际或 domain。

### 14.2 普通家庭拖拽

- 只允许同代、同根族区间内排序；
- 同一行只有一个家庭时也可写入网格列，因此根家庭和孤立家庭仍可单独移动；
- 相邻家庭实时漂移让位；
- 拖到其他根族或桥接区域时显示无效落点并回弹；
- 不改变根签名、代际或关系。

### 14.3 跨根家庭拖拽

- 只允许在所属桥接带或桥接岛的同代行内排序；
- 每个跨根家庭拥有独立网格列偏好，移动一个家庭不会把整个桥接域一起拖走；
- 联合后代在放下后重新居中；
- 不能拖进单根区域。

### 14.4 根家庭拖拽

- 根家庭与其他家庭一样只移动自身家庭单元；
- 后代家庭不跟随拖动，放下后只重新计算受影响的行、分支居中和线路；
- 按住 `Ctrl`（macOS 同时支持 `Command`）拖动根家庭时切换为根族整体模式；
- 整体模式下根域内所有家庭同步预览位移，放下后写入 `rootOrders`，相关桥接区域按新的来源根重心重新计算；
- 快捷键状态在 pointer down 时锁定，一次拖动过程中不切换模式，避免误操作。

### 14.5 偏好失效

关系变化后，偏好必须按稳定 ID 重新校验：

- 已删除家庭从偏好中移除；
- 根签名改变的家庭迁移到新 domain；
- 跨根变为同根时删除对应 bridge 偏好；
- 同根变为跨根时不沿用旧 root row 偏好；
- 无法满足硬约束的偏好被忽略并报告诊断，不能修改事实迎合偏好。

## 15. 恢复默认布局

“恢复默认布局”需要清除：

- `rootOrders`；
- 根族内部 `rowOrders`；
- `bridgeOrders`；
- 当前 UI 会话中的画布缩放和平移。

然后：

1. 以当前关系事实重新识别根家庭和根签名。
2. 不携带旧场景执行完整默认布局。
3. 取消尚未完成的普通家庭、桥接家庭和根家庭单元拖拽。
4. 只接受最新异步布局请求。
5. 恢复 100% 缩放。
6. 有有效 `viewpointId` 时聚焦人物卡。
7. 否则聚焦全部可见家族树的整体包围框中心。

## 16. 辅助关系

- 历史配偶、次要父母、继亲和干亲只进入辅助路由层。
- 打开或关闭辅助层不得重新识别根、改变 domain 或移动主体家庭。
- 辅助线可以跨越根族留白，但必须使用独立线型和低视觉权重。
- 辅助线不能与主体线共享线段，也不能造成伪汇合。
- `childLayoutAssignments` 改变时属于主体语义变化，允许重新计算根签名和 domain。

## 17. 增量稳定性

### 17.1 上一个场景匹配

新旧根族通过以下优先级匹配：

1. 相同稳定根 ID；
2. 最大成员重叠；
3. 相同根家庭后代重叠；
4. 稳定 ID 平局处理。

匹配成功后继承：

- 根族主色；
- 根族左右相对顺序；
- 未受影响家庭的网格位置；
- 家庭线路变体色。

### 17.2 局部变化

- 根族内部新增成员只扩宽该根族和直接相关分支。
- 新增跨根婚姻只创建或扩宽相关桥接区域。
- 根动态上移优先保留原根族 footprint。
- 不相关根族不得因为局部编辑发生无理由翻转。

## 18. 异常与边缘场景

### 18.1 多个不相连分量

每个分量独立执行根族布局，再按 `rootMemberId`、分量规模和稳定 ID 打包。分量之间使用比根族间距更大的留白。

### 18.2 谱系回接与同人多路径

人物只渲染一次。根签名使用集合并集。位置由所有主体入边的加权重心决定，但必须留在所属 domain。

### 18.3 关系环

检测到父母关系环时：

- 不修改源数据；
- 产生明确诊断；
- 为布局确定性地暂时断开最低优先级主体边；
- 使用安全场景保证卡片不重叠；
- 不把环中的每个人错误识别为根。

### 18.4 根过多

根色数量超过推荐调色板时，先保证空间留白和色条明度差，再复用色相。相邻根族不得使用近似色。

### 18.5 空树和失效视角

空树聚焦画布中心。`viewpointId` 找不到对应卡片时回退到全部场景中心。

### 18.6 桥接区域过密

先扩宽桥接区，再增加路由子通道，最后才允许更多点交叉。不得通过共线或穿卡压缩桥接区。

## 19. 场景输出与诊断

`LayoutScene` 增加布局语义元数据：

```ts
interface LayoutScene {
  rootDomains: PlacedRootDomain[]
  bridgeDomains: PlacedBridgeDomain[]
  rootSignaturesByUnitId: Record<string, RootSignature>
  units: PlacedFamilyUnit[]
  cards: PlacedPersonCard[]
  hubs: PlacedUnionHub[]
  routes: RoutedFamilyEdge[]
  bounds: Rect
  diagnostics: LayoutDiagnostic[]
}
```

新增诊断至少包括：

- `ROOT_DOMAIN_INTERLEAVING`
- `UNIT_OUTSIDE_DOMAIN`
- `INVALID_ROOT_SIGNATURE`
- `BRIDGE_DOMAIN_OVERFLOW`
- `CROSS_DOMAIN_ROUTE_INTRUSION`
- `STALE_LAYOUT_PREFERENCE`

生产环境不得静默丢失人物或线路。硬约束失败时使用非重叠安全场景并显示诊断。

## 20. 性能目标

默认目标为 500 位成员：

- 根识别和签名传播接近 `O(V + E)`；
- 根族关系图只聚合跨根家庭；
- 根族内部布局按 domain 独立计算；
- 单个根族变化不重新计算无关根族内部坐标；
- 路由按 domain 并行分区，最后只处理跨 domain 路线；
- 交互预览不得每帧运行完整布局。

拖拽期间只计算目标行和 domain 邻居漂移；放下后再执行受影响 domain 的正式布局。

## 21. 验收标准

1. 同一根族在所有代际中保持连续，其他根族不能插入。
2. 根族之间仅通过留白、根色和根家庭强化即可明显区分。
3. 根家庭没有标题、背景或根图标。
4. 外来未展开配偶不会错误产生新根。
5. 双根家庭进入桥接带，多根密集关系进入桥接岛。
6. 联合后代保持多根签名，不强制归入单侧。
7. 同根跨分支婚姻留在根族内部。
8. 每个人物卡只出现一次。
9. 不同家庭线路不得共线、穿卡或形成伪汇合。
10. 跨根双方父母线分别连接各自人物卡顶部。
11. 辅助关系开关不改变主体坐标。
12. 普通家庭、跨根家庭和根家庭单元拖拽均不能改变关系事实。
13. 恢复默认布局清除三类排序偏好并正确聚焦。
14. 补录更早祖辈后根动态上移，但原根族颜色和大部分位置保持稳定。
15. `rootMemberId`、`viewpointId` 和根族身份互不混用。
16. 500 位成员场景能够完成布局并通过硬约束验证。

## 22. 测试策略

### 22.1 纯函数单元测试

- 根家庭识别；
- 外来配偶继承；
- 根签名集合传播；
- 同根跨分支婚姻；
- A×B、A×B×C 联合分支；
- 收养、继亲、历史配偶和干亲投影；
- 根族关系图排序；
- 连续列区间分配；
- 根动态上移与颜色继承。

### 22.2 几何属性测试

- domain 不交叠；
- 单根 unit 全部位于所属 domain；
- 同代单根 unit 连续；
- 卡片和家庭单元不重叠；
- 不同 route owner 不共享线段；
- 路线不穿越无关 domain；
- 交叉不形成伪汇合；
- 相同输入确定性输出。

### 22.3 组件测试

- 根家庭隐式强化；
- 根色条和家庭线路变体；
- 跨根双色夫妻轴；
- 普通、跨根和根家庭单元的拖拽范围、`Ctrl` 根族整体模式与无效落点；
- 单家庭网格列偏好与相邻单元漂移；
- 辅助关系开关不移动主体；
- 恢复默认布局清除全部偏好并聚焦。

### 22.4 回归场景

- 两个根族通过一个家庭连接；
- 同代多个 A×B 家庭；
- A×B、B×C 链式桥接；
- A×B、B×C、A×C 三角桥接岛；
- 同根堂亲婚姻；
- 多段配偶和历史关系；
- 收养与生父母并存；
- 多个不相连家族分量；
- 补录更早祖辈；
- 500 人多根压力场景。

## 23. 实施边界

本设计需要重构布局语义和 domain 分配，但不需要重写人物关系编辑器或事实存储。推荐分阶段替换：

1. 增加根识别、根签名和诊断，不改变坐标。
2. 增加根族连续区间和内部布局。
3. 增加桥接带、桥接岛和根族感知路由。
4. 增加两级颜色与根家庭强化。
5. 增加家庭单元顺序、网格列偏好和完整恢复默认布局。
6. 完成增量稳定性、真实边缘场景和 500 人性能验证。

每个阶段必须保持可运行、可验证，并继续通过既有线路安全不变量。
